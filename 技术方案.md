# RAG 企业级知识库问答系统 — 技术方案

## 一、项目概述

基于 LangChain + 阿里云百炼 API 构建电商商品知识库问答系统，支持多用户、多会话、知识库管理和引用溯源。

---

## 二、为什么选择阿里云百炼（Bailian / DashScope）

阿里云百炼是阿里推出的一站式大模型服务平台，核心优势：

| 优势 | 说明 |
|------|------|
| **中文能力强** | 通义千问（Qwen）系列是国内中文能力最顶尖的开源模型之一，电商场景的中文理解和生成远超通用英文模型 |
| **兼容 OpenAI 协议** | 百炼提供 OpenAI 兼容接口，只需要改 `base_url` 和 `api_key`，代码写法跟调用 OpenAI 一模一样，LangChain 无缝接入 |
| **一套 API 全覆盖** | 同一个平台同时提供 LLM 对话、Embedding 向量化、Rerank 重排序三种能力，不需要对接多个厂商 |
| **成本可控** | qwen-plus 模型性价比极高，同时有免费额度（Embedding 送 100 万 Token） |
| **国内部署低延迟** | 北京/新加坡/香港多地域，国内访问延迟低，不走海外网络 |

## 三、技术栈选择与解释

### 后端核心

| 技术 | 选型 | 为什么 |
|------|------|--------|
| **编程语言** | Python 3.12+ | LangChain 原生 Python 生态最成熟；AI/ML 领域事实标准 |
| **Web 框架** | **FastAPI** | 异步高性能，原生 OpenAPI/Swagger 文档，自带 Pydantic 参数校验，比 Flask 快一个数量级，适合企业级 API |
| **LLM 框架** | **LangChain** + LangGraph | LangChain 负责 RAG 链编排（文档加载→分割→向量化→检索→生成）；LangGraph 负责有状态的多轮对话流程 |
| **LLM 服务** | **阿里云百炼 DashScope**（OpenAI 兼容协议） | 主力模型 qwen-plus，复杂任务 qwen-max。通过 `langchain_openai.ChatOpenAI` 接入，改 base_url 即可 |
| **Embedding 模型** | **text-embedding-v4**（百炼） | 最新 Qwen3-Embedding，支持 100+ 语种，维度可选 64~2048，RAG 场景推荐 1024 维。通过 `langchain_openai.OpenAIEmbeddings` 接入 |
| **Rerank 模型** | **qwen3-rerank**（百炼） | 支持最多 500 篇文档重排序，单篇 4000 Token，支持 OpenAI 兼容接口。**注意：旧的 gte-rerank 已于 2026 年 5 月下线** |
| **向量数据库** | **Milvus** (生产) / **Chroma** (开发) | Chroma 零配置适合本地开发调试；Milvus 支持十亿级向量检索、IVF_FLAT/HNSW 索引，适合企业级生产 |
| **关系数据库** | **PostgreSQL** | 存储用户、会话、消息等结构化数据；支持 JSONB 字段灵活扩展 |
| **缓存** | **Redis** | 缓存热点知识库片段、会话状态、限流计数；大幅降低 LLM 调用延迟 |
| **异步任务** | **Celery** + Redis Broker | 文档上传后异步处理（解析→切片→向量化→入库），避免阻塞用户请求 |
| **对象存储** | **MinIO** (S3 兼容) | 存储上传的原始文档（PDF/Word/TXT），自托管，S3 兼容可无缝迁移到云 |

### 百炼 API 接入方式

由于百炼全面支持 OpenAI 兼容协议，**不需要安装额外的 LangChain 集成包**，直接用 `langchain-openai` 即可，所有接口统一。这也是目前 LangChain 社区推荐的最佳实践：

```python
# LLM 对话模型
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(
    model="qwen-plus",                     # 或 qwen-max / qwen-turbo
    openai_api_key="sk-xxx",               # 百炼 API Key
    openai_api_base="https://dashscope.aliyuncs.com/compatible-mode/v1",
    temperature=0.1,
)

# Embedding 向量化模型
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(
    model="text-embedding-v4",
    openai_api_key="sk-xxx",
    openai_api_base="https://dashscope.aliyuncs.com/compatible-mode/v1",
    dimensions=1024,  # 推荐 1024 维，平衡精度和性能
)
```

| 百炼模型 | 用途 | 说明 |
|----------|------|------|
| `qwen-plus` | 主力 LLM（推荐） | 效果与成本最佳平衡，适合 RAG 问答 |
| `qwen-max` | 复杂任务 LLM | 高难度推理、多步分析 |
| `qwen-turbo` | 轻量 LLM | 问题改写等简单任务，成本最低 |
| `text-embedding-v4` | 文本向量化 | 最新版本，维度可选，每次最多 10 条 |
| `qwen3-rerank` | 检索结果重排序 | 精排召回结果，提升 RAG 精度 |

## 五、数据库设计

### PostgreSQL 核心表

#### 1. users（用户表）
```
id, username(唯一), password_hash, email, role(admin/user),
is_active, created_at, updated_at
```

#### 2. conversations（会话表）
```
id, user_id(FK), title, created_at, updated_at
```
- 每个用户可创建多个会话，每个会话对应一个独立的话题

#### 3. messages（消息表）
```
id, conversation_id(FK), role(user/assistant), content,
citations(JSONB，引用的知识库片段), created_at
```
- `citations` 字段存储 JSON：`[{"doc_id": "xxx", "chunk_id": "yyy", "text": "...", "score": 0.92}]`

#### 4. documents（知识库文档表）
```
id, filename, file_type, file_size, file_path(MinIO),
status(processing/done/error), chunk_count, uploaded_by(FK),
created_at, updated_at
```

#### 5. document_chunks（文档切片表）
```
id, document_id(FK), chunk_index, content, vector_id(Milvus),
token_count, created_at
```
- `vector_id` 关联 Milvus 中的向量，用于删除/更新时同步

### Milvus Collection

```
collection: knowledge_base
fields: id(PK), embedding(1024维向量), doc_id, chunk_id, content(文本),
        metadata(JSON: 文件名/类型/页码等)
index: HNSW（小数据量 <100万） 或 IVF_FLAT（大数据量）
```

### Redis 缓存策略

| Key | 内容 | TTL |
|-----|------|-----|
| `session:{token}` | 用户会话信息 | 24h |
| `rate:{user_id}` | API 调用频率计数 | 1min |
| `kb:hot:{query_hash}` | 热点问题向量检索结果 | 10min |
| `embedding:{hash}` | 常用文本的 Embedding 缓存 | 1h |

## 六、RAG 核心流程

```
用户提问
  │
  ▼
┌──────────────────┐
│ 1. 问题改写       │  qwen-turbo 结合历史对话改写问题（去指代消解）
│   (Query Rewrite) │  如 "它多少钱" → "iPhone 15 多少钱"
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 2. 混合检索       │  向量检索(Milvus) + 关键词检索(BM25)
│   (Hybrid Search) │  各取 top-10，合并去重得 ~20 条候选
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 3. 重排序         │  qwen3-rerank 对候选精排
│   (Rerank)       │  取 top-5 送入 LLM
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 4. Prompt 组装    │  System: "基于以下知识库回答...{chunks}"
│                   │  插入检索片段 + 对话历史
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 5. LLM 生成       │  qwen-plus 流式生成带引用的回答
│   (Generation)   │  要求模型标注引用编号 [1] [2]
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 6. 后处理         │  解析引用标记 → 关联实际知识库片段
│   (Post Process) │  格式化为 Markdown + 引用区块返回前端
└──────────────────┘
```

### 为什么需要混合检索 + 重排序？

**单靠向量检索不够**，举个电商例子：

> 用户问："SKU-A1234 支持快充吗？"

- **向量检索**擅长语义："充电速度"、"续航"相关片段能召回，但可能漏掉精确的型号匹配
- **BM25 关键词检索**擅长精确匹配："SKU-A1234" 直接命中商品规格表
- 两者各取 top-10，合并后约 15-20 条候选（高召回）
- **qwen3-rerank 精排**：对这 20 条重新打分，取最相关的 5 条（高精度）

这样既不会漏掉结果（高召回），也不会给 LLM 塞太多无关内容（高精度）。

### 关键优化点

1. **混合检索（Hybrid Search）**：向量检索 + BM25 关键词检索，召回率提升 15-30%
2. **重排序（Rerank）**：qwen3-rerank 精排，初检 20 条 → 精排 5 条，控制 Token 消耗
3. **查询改写（Query Rewrite）**：多轮对话中用轻量模型 qwen-turbo 改写指代不清的问题
4. **结果缓存**：相同/相似问题直接返回缓存结果
5. **文档分块策略**：
   - 商品描述用固定大小分块（chunk_size=512, overlap=50）
   - 商品参数表用 Markdown 标题分割
   - 保留文档元数据（文件名、所属品类、更新时间）

## 七、API 设计

### 认证模块 `/api/auth`
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/register` | 用户注册 |
| POST | `/login` | 用户登录，返回 JWT |
| POST | `/change-password` | 修改密码（需登录） |
| GET  | `/me` | 获取当前用户信息 |

### 对话模块 `/api/chat`
| 方法 | 路径 | 说明 |
|------|------|------|
| GET  | `/conversations` | 获取当前用户的所有会话列表 |
| POST | `/conversations` | 创建新会话 |
| DELETE | `/conversations/{id}` | 删除会话 |
| GET  | `/conversations/{id}/messages` | 获取会话历史消息 |
| POST | `/conversations/{id}/query` | 发送消息，SSE 流式返回 |

### 知识库管理模块 `/api/knowledge`（仅管理员）
| 方法 | 路径 | 说明 |
|------|------|------|
| GET  | `/documents` | 文档列表（分页） |
| POST | `/documents/upload` | 上传文档（PDF/Word/TXT/CSV/Markdown） |
| DELETE | `/documents/{id}` | 删除文档（同步删除向量） |
| GET  | `/documents/{id}/chunks` | 查看文档切片详情 |
| POST | `/documents/reprocess` | 重新处理失败的文档 |
| GET  | `/stats` | 知识库统计 |

## 十三、验证方式

1. `docker compose up -d` 一键启动所有服务
2. 浏览器访问 `http://localhost:3000`，注册新用户
3. 用 `admin / 123456` 登录，上传测试文档（PDF/Word/TXT）
4. 等待 Celery 处理完成（文档状态变为"done"）
5. 用普通用户登录，创建会话并提问，验证引用是否正确展示
6. 退出重新登录，验证历史会话是否保留
7. 尝试用普通用户访问 `/admin/knowledge`，验证被拦截
8. 用 Locust 压测 `/api/chat/query` 接口（模拟 50 并发）
